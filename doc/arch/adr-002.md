# ADR-002: 支持双目录 skill 加载：内置 skills 与用户自定义 skills 分离

- **Date**: 2026-05-06
- **Status**: Accepted
- **Deciders**: arcanexis
- **Tags**: 

## Context

ADR-001 已将 skills/ 迁移至 backend/alice/skills/（后端包内，受版本控制）。但仓库原先 18 个 skill 全部迁入版本控制导致了两个问题：

- 仓库臃肿：18 个 skill 带大量 schema/资源文件（docx 的 ooxml schemas 就有几十个 .xsd），git 历史膨胀。
- 语义混淆：真正需要版本控制的只有 skill-creator（元 skill，用户用它能创建新 skill）；其余 skill（akshare、docx、brand-guidelines、web-artifacts-builder 等）本质是"用户按需安装的运行时能力"，不应发版到仓库里。

同时，`Settings.skills_dir` 当前只支持单一路径。如果仓库只保留 skill-creator，用户从社区安装的 skill 放在 `.alice/skills/`（用户私有目录，被 .gitignore 忽略），就没有机制同时加载两个来源。

## Decision

将 skill 加载从单目录模型改为双目录加载模型：

- **内置目录**：`backend/alice/skills/` — 仓库版本控制，当前仅含 `skill-creator`。作为系统预置能力基线。
- **用户目录**：`.alice/skills/` — 用户私有，不受 git 管理。系统启动时若不存在则自动创建。用户通过 skill-manager 等工具安装的 skill 落在此处。
- **加载顺序**：先加载内置目录，再加载用户目录。同名 skill 以用户目录为准（用户可覆盖内置 skill）。
- **不变部分**：skill 内部结构（SKILL.md + 可选 scripts/）、SKILL.md 格式规范、容器内 `/app/skills/` 路径语义均不变。

实现方式：
- `Settings` 新增 `skills_user_dir: str = '.alice/skills'`，与现有 `skills_dir` 并列。
- 所有 loader 接口从 `skills_dir: str` 改为 `skills_dirs: list[str | Path]`，内部按序遍历、后覆盖前。
- `FileRepository` 支持多目录查找。
- `DockerConfig.default_mounts` 新增 `.alice/skills/` 挂载。
- `ContainerManager._ensure_mount_directories` 确保 `.alice/skills/` 存在。
- Runtime registry bootstrap 时传入 `[settings.skills_dir, settings.skills_user_dir]`。

## Consequences

- 积极：仓库信噪比大幅提高（backend/alice/skills/ 仅 1 个内置 skill，约 900 行）；用户可按需安装社区 skill 无需改仓库；版本控制和用户定制之间的边界清晰。
- 消极：从旧版单目录升级到双目录时，用户本地若曾直接在 `skills/` 下添加过自定义 skill，需要手动迁移到 `.alice/skills/`。新用户无此负担。
- 中性：同名覆盖规则意味着用户可以"fork"内置 skill 的行为——这是能力而不是 bug，但需要在文档中说明；开发者调试时需确认"当前生效的是哪个目录的版本"。

## Alternatives Considered

- **保持单目录，把全部 skill 留在仓库**：仓库持续臃肿，每个 skill 更新都需一次 commit，与"skill 是用户运行时能力"的定位冲突。排除。
- **单目录 + 符号链接**：把用户 skill 链接到仓库 skills/ 下，平台兼容性差（Windows），且 git 行为不确定。排除。
- **多目录但用户优先（与选定方案相反的顺序）**：违背"内置 skill 是基线、用户可覆盖"的直觉模型，且容器挂载场景下更难处理。
