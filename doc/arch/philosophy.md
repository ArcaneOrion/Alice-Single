# Alice Philosophy

## 初心

Alice 是一个面向个人 AI 的自迭代实验场。

它以个人助理作为第一真实应用场景，通过长期对话、任务执行、记忆演化、代码修改、验证反馈和用户校正，探索 agent harness 如何随模型能力和用户理解一起演化。

Alice 的目标不是先做成通用 Agent 平台，也不是做一个普通聊天助理，而是在真实个人场景中持续观察、设计、验证和改进 AI 行为边界。

## 归档原因

Alice 归档封存为 v0 记忆，不再作为后续主要管理对象。

原因不是旧项目失败，而是新的方向已经超出原地重构的范围。新方向的核心是通过 API 对自身 harness 进行受控自我演化的 Agent 框架；旧 Alice 的实际代码形状已经带有较多 runtime / personal assistant app 的历史假设。

Alice 保留为前史、经验样本和对照组。后续新框架由 Praxis 承载。

## 核心定位

Alice 不是单个助理应用，而是让个人 AI 助理逐渐变好的实验环境。

个人助理是第一负载，因为它天然包含真实需求、长期记忆、偏好对齐、学习陪伴、项目协作、行动提醒和代码实战。没有真实负载，自迭代会变成空转；只按助理应用设计，又会把更深的实验系统压扁成聊天功能。

## 设计原则

1. 真实场景优先于玩具 benchmark。
2. 观察、记录、评估和回放是自迭代的地基。
3. 每次 prompt、memory、skill、tool policy、workflow 或架构调整，都应能被理解为一次实验。
4. Harness 应该提供环境、边界、观测和反馈，而不是替模型写死行为。
5. 随着模型能力增强，外部框架应倾向更简单、更少假设、更少硬编码。
6. 架构设计应服务自迭代循环，而不是围绕功能清单堆叠模块。
7. 个人助理经验既是使用价值，也是用于探索 agent 边界的实验数据。

## 最小自迭代循环

Alice 的核心循环暂定为：

```text
用户/环境输入
  -> agent 行为
  -> trace / memory / validation
  -> reflection / evaluation
  -> 设计变更
  -> 验证
  -> 固化到代码、文档、ADR、Spec 或记忆
```

后续架构讨论应围绕这个循环审计：某个模块、抽象或流程是否让这个循环更清楚、更可靠、更可演化。

## 非目标

当前阶段不以通用 Agent 平台为第一目标。

当前阶段不以多端产品化、复杂网关、完整插件市场或通用工作流引擎为第一目标。

当前阶段不追求一次性设计出最终架构。Alice 的架构应允许随着真实使用、模型变化和用户理解增长而演化。

## 待讨论问题

- 自迭代循环中哪些数据必须进入 trace，哪些只进入日志或记忆。
- evaluation 应该从人工审计、规则检查、任务回放还是自动评分开始。
- 个人助理、代码协作、学习陪伴、行动工程等场景如何形成统一实验接口。
- 现有 gateway、workflow、bridge、skills、memory、logging、spec-vc 等模块应保留、冻结、降级还是重构。
- 哪些设计原则应该升级为硬约束，哪些只作为阶段性审美偏好。

## 使用方式

这份文档不是实现计划，也不是固定蓝图。它是 Alice 后续设计和重构时的初心锚点。

当新的架构想法出现时，先问：

1. 它是否增强了 Alice 的自迭代循环。
2. 它是否来自真实个人场景中的压力。
3. 它是否让 harness 更清楚，而不是更沉重。
4. 它是否能被观察、验证，并在必要时回滚。
