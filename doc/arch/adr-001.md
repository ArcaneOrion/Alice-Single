# ADR-001: 目录结构重组：skills 归入后端包、Dockerfile 建 docker/ 目录收容

- **Date**: 2026-05-06
- **Status**: Accepted
- **Deciders**: arcanexis
- **Tags**: 

## Context

仓库根目录散落着多种性质不同的资产：

- `skills/`：后端 Python agent 的运行时能力模块（SKILL.md + 可选脚本），由 Python loader 在 bootstrap 时扫描加载。
- `Dockerfile.sandbox`：后端容器执行后端用的镜像定义。
- `prompts/`：运行时提示词模板源。
- `frontend/`、`backend/`：前后端代码包。

这种排布让"哪些是后端运行时构成、哪些是用户运行时配置、哪些是部署/打包资产"边界模糊。`skills/` 实质是后端包的能力库，但放在根目录看起来像独立模块；`Dockerfile.sandbox` 与 `docker-compose.yml`（如有）等部署文件分散在根。`.alice/` 目录被 `.gitignore` 忽略，是用户私有运行时配置，不能用来收容这些需要版本控制的资产。

## Decision

将后端运行时构成的资产归入后端包，将容器镜像定义集中到 `docker/`：

- `skills/` → `backend/alice/skills/`（`git mv` 保留历史）
- `Dockerfile.sandbox` → `docker/Dockerfile.sandbox`
- `prompts/` 保持不动（提示词模板源是跨前后端的运行时输入，不属于后端包内构成）

同步更新所有硬编码路径：`Settings.skills_dir` 默认值、`DockerConfig.default_mounts` 的宿主机源路径、`DockerConfig.dockerfile_path`、skills loaders/repository/cache 的默认参数、`prompts/04_tools.xml` 的路径描述、`README.md` 目录结构。容器内路径语义不变（仍为 `/app/skills/`），仅调整宿主机挂载源。`backend/pyproject.toml` 的 mypy `exclude` 增加 `backend/alice/skills/`，避免无 `__init__.py` 的子目录触发命名空间冲突。

## Consequences

- 积极：后端包 self-contained，`backend/alice/` 同时包含业务代码与技能库；根目录信噪比提高；docker/ 集中所有镜像定义，未来追加 dev / prod / sandbox 等多镜像更容易扩展。
- 消极：用户本地 `.alice/config.json` 中若覆写了 `skills_dir` 指向旧路径 `skills/`，升级时需要手动改成 `backend/alice/skills/`，无自动迁移。
- 中性：`backend/alice/skills/` 子目录无 `__init__.py` 意味着不可被 import 为 Python 模块——这是刻意的（skills 由 loader 以文件方式扫描，不通过 import 机制），但需要 mypy `exclude` 配合。

## Alternatives Considered

- **保持根 `skills/` 不动**：根目录持续臃肿，且 skills 与后端包的语义关系不显式。排除。
- **将 `skills/` 放入 `.alice/skills/`**：`.alice/` 已被 gitignore，会失去版本控制。排除（但本仓库的"内置 skill"与"用户自定义 skill"的二元划分由 ADR-002 在此基础上引入）。
- **把 `Dockerfile.sandbox` 留在根**：容器定义未来不会只有一个（dev/prod/sandbox），过早集中比未来再迁更省事。排除。
