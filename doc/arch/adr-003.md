# ADR-003: TUI 错误码分级展示与后端错误分类体系

- **Date**: 2026-05-08
- **Status**: Accepted
- **Deciders**: arcanexis
- **Tags**: 

## Context

当前 TUI 错误展示存在两个问题：

1. **错误不分级**：桥接协议错误（`_send_error` 走 stdout JSON）、stderr 日志泄露、前端内部错误全部以相同 `⚠️` 展示给用户，无法区分"会话终止"、"请求失败但可重试"、"不影响功能的警告"。
2. **stderr traceback 泄露**：`openai_provider.py` 在 API 失败时打 `logger.error(..., exc_info=True)`，完整 Python traceback 通过 stderr 管道进入前端。前端的 `is_actionable_stderr_error` 函数匹配 `[ERROR]` 级别和 `Traceback` 关键字，将 traceback 的多行代码上下文逐行作为 actionable error 显示在 TUI，严重污染界面。

桥接协议 schema（`protocols/bridge_schema.json`）已定义 `ErrorMessage.code` 为可选 string 字段，但 Rust 前端 `BridgeMessage::Error` 未反序列化该字段，Python 后端 5 个 `_send_error` 调用点全部传 `code=""`。

## Decision

建立 3 级错误码体系，实现端到端的错误分级展示和 stderr 整治：

**错误码**：
- `fatal` — 会话终止（初始化失败、主循环崩溃）
- `error` — 当前请求失败，session 仍存活（API 调用失败、回调异常、用户输入处理异常）
- `warn` — 不影响功能的警告（预留）

**Rust 前端**：
- `BridgeMessage::Error` 改为 `{ content, code: Option<String> }`，`#[serde(default)]` 确保缺失 code 时反序列化为 None，向后兼容旧版后端。
- `App::add_error(content, code)` — fatal → "FATAL" 前缀、error → "ERROR" 前缀、warn → "WARN" 前缀。
- `is_actionable_stderr_error` 收紧：仅 `[FATAL]` 或 `[CRITICAL]` 级别的 Python 日志行才作为 actionable error；`[ERROR]` 和 `Traceback` 行不再展示（仅写入 runtime_log）。非日志格式的 stderr 输出（解释器崩溃等）仍全量 actionable。

**Python 后端**：
- 定义 `ERROR_FATAL` / `ERROR_RECOVERABLE` / `ERROR_WARN` 字符串常量。
- `_send_error(content, code)` — 5 个调用点：初始化失败 → fatal、主循环异常 → fatal、回调异常 → error、Agent 未初始化 → error、用户输入处理异常 → error。
- `openai_provider.py` 两处 `exc_info=True` → `exc_info=logger.isEnabledFor(logging.DEBUG)`，traceback 仅在 DEBUG 级别时输出到 stderr。

## Consequences

- 积极：用户可区分错误严重程度；TUI 不再被 traceback 污染；stderr 完整内容仍通过 runtime_log 保留不丢失调试信息；新增 code 字段向后兼容（schema 定义为 optional、serde 缺省 None）。
- 消极：`[ERROR]` 级别的 stderr 日志不再展示在 TUI——用户需要主动查看 `runtime_log` 才能看到完整错误细节。
- 中性：`fatal` 和 `error` 的分界依赖调用点的判断——如果异常在 `_on_message_received` 被捕获，它被标记为 error（可恢复），但若同一异常逃逸到主循环，则标记为 fatal。这是正确的语义区分而不是 Bug。

## Alternatives Considered

- **保持原有 `[ERROR]` 匹配，仅去掉 `exc_info=True`**：能消除 traceback 但 `[ERROR]` 级别的单行日志仍会出现在 TUI，且无错误分级能力。排除。
- **在 Python 端将所有 logger stderr 输出重定向到文件**：需要改动 stderr 整体架构（当前设计是"stdout=JSON 协议、stderr=日志"），破坏调试便利性，改动面大。排除。
- **在 Rust 端给 `Message` 增加 `error_style` 字段实现颜色渲染**（而非纯文本前缀）：需要动 ratatui Span 渲染链，增加复杂度但收益有限。可作为后续优化，本轮采用纯文本前缀方案。
