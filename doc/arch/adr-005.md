# ADR-005: Alice 重定位为个人 AI 自迭代实验场

- **Date**: 2026-05-26
- **Status**: Accepted
- **Deciders**: arcanexis
- **Tags**: philosophy, architecture, self-iteration

## Context

Alice 最初不是单纯为了实现一个聊天助理或通用 Agent 平台，而是在个人使用场景中探索 Agent 能力边界、自迭代方式和 harness 设计的一次长期实验。

随着项目推进，仓库逐步长出了 Rust TUI、Python 后端、bridge 协议、workflow、memory、skills、docker sandbox、gateway、logging、ADR/Spec 等多类模块。这些模块各自有局部合理性，但项目的上位哲学没有被显式记录，导致后续设计容易被局部工程问题牵引。

当前出现的架构不适感主要不是某个模块实现错误，而是项目实际形状与原始意图之间出现漂移：Alice 正在从"个人 AI 的自迭代实验场"滑向"功能较多的 Agent runtime / personal assistant app"。如果不先记录初心和判断标准，后续重构容易继续围绕模块做加减法，而不是围绕自迭代循环重新审计。

后续讨论进一步确认：新的方向已经不再适合在 Alice 现有代码库中原地重构。新方向的核心是通过 API 对自身 harness 进行受控自我演化的 Agent 框架，而旧 Alice 的实际形状已经承载了较多 runtime / personal assistant app 的历史假设。继续在旧仓库上推进，会让新哲学被旧抽象和旧模块牵引。

## Decision

我们将 Alice 的上位定位记录为：Alice 是一个面向个人 AI 的自迭代实验场。

Alice 以个人助理作为第一真实应用场景，通过长期对话、任务执行、记忆演化、代码修改、验证反馈和用户校正，探索 agent harness 如何随模型能力和用户理解一起演化。

我们新增 `doc/arch/philosophy.md` 作为项目哲学与设计初心锚点。该文档不是具体重构计划，而是后续架构讨论、模块取舍和 ADR 决策的上位审计标准。

Alice 作为 v0 原型归档封存，留作项目记忆、经验样本和历史对照，不再作为后续主要管理和演化对象。新的 Agent 框架方向将以新项目 Praxis 承载。

具体技术架构将在 Praxis 中重新设计，不在 Alice 中继续展开。

## Consequences

- 积极后果:
  - 保留 Alice 作为 v0 经验样本和历史对照。
  - 新项目可以避开旧 runtime / app 形状的惯性，从 harness 受控自我演化的哲学重新开始。

- 消极后果:
  - Alice 不再作为主要项目继续演进，旧代码中的部分经验需要未来按需迁移或重做。

- 中性后果:
  - Alice 仓库保留为历史记录，不删除。
  - 本 ADR 不对 Praxis 的技术细节做承诺。

## Alternatives Considered

1. **只做局部工程减法**：可以缓解当前复杂度，但无法解释为什么这些复杂度是不优雅的，也无法防止未来再次漂移。
2. **把 Alice 定位为普通个人助理应用**：能带来更直接的产品目标，但会压扁自迭代、实验、trace、evaluation 和 harness 演化这些更深层目标。
3. **把 Alice 定位为通用 Agent 平台**：扩展性更强，但会过早平台化，容易继续引入 gateway、多端、多租户、插件市场等当前阶段不必要的复杂度。
4. **在旧 Alice 上原地重构为新框架**：可以复用已有代码，但旧架构的中心对象已经偏向 runtime / app，容易让新项目继续继承不合适的抽象。排除。
