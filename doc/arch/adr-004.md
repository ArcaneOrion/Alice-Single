# ADR-004: ChatMessage reasoning_content 兼容：修复 thinking mode 多轮对话消息回传缺失

- **Date**: 2026-05-18
- **Status**: Accepted
- **Deciders**: arcanexis
- **Tags**: bug-fix, compatibility, thinking-mode

## Context

GLM thinking mode 要求多轮对话中 assistant 消息必须回传 `reasoning_content` 字段，否则 API 返回 400 错误。当前代码在提取 thinking 方面完整（`StreamChunk`/`ChatResponse` 支持 6 种字段名），但在构造消息历史时全部丢弃——`ChatMessage` 无此字段，`to_dict()` 不输出，`from_dict()` 不读入，所有 assistant 消息构造入口均不传递 thinking 内容。

数据断流点：
| 位置 | 问题 |
|------|------|
| `ChatMessage` 模型 | 无 `reasoning_content` 字段 |
| `ChatMessage.to_dict()` | 不输出 `reasoning_content` |
| `ChatMessage.from_dict()` | 不读入 `reasoning_content` |
| `ChatService.add_assistant_message()` | 不接受/传递 thinking |
| `FunctionCallingOrchestrator` | 构造 assistant_message 时不带 thinking |

## Decision

在 `ChatMessage` 数据结构中新增 `reasoning_content: str | None = None` 可选字段，并在序列化时条件性输出（有值时才包含）。以最小侵入方式将 thinking 内容从响应层传递到消息历史层。

设计原则：
- `None` 或空字符串时不输出该字段（变更前行为完全不变）
- 非空时输出 `reasoning_content`，GLM thinking mode 可正确接收
- OpenAI 等忽略未知字段的 API 不受影响，无需 capability gate

## Consequences

- **积极后果**:
  - GLM thinking mode 多轮对话不再报 400
  - 所有 thinking mode 模型（deepseek-r1、QwQ 等）均受益，reasoning 内容可正确回传
  - 变更完全向后兼容，`reasoning_content=None` 时行为与变更前一致

- **消极后果**:
  - `reasoning_content` 在长推理过程中可能很长，增加 token 消耗（但目前仅在模型原生返回时才填充，不由用户控制）

- **中性后果**:
  - 消息历史的 API dict 体积可能增大，但这是 thinking mode 的固有代价

## Alternatives Considered

1. **在 Provider 层做字段名适配**：将内部 `thinking` 统一映射为各 API 要求的字段名——过度设计，当前只需一个 `reasoning_content` 字段即可覆盖主流 thinking mode 模型
2. **增加 `supports_thinking` capability gate**：仅在模型声明支持 thinking 时才发送 `reasoning_content`——不必要，模型会自行忽略未知字段，gate 增加了维护负担
3. **在桥接层做但不在消息层做**：无法解决多轮对话的根因问题，thinking 需要在消息历史中持久回传
